# Linear 任务创建指南

最后更新：2026-06-28

这份文档回答两个问题：

1. 什么时候应该在 Linear 上创建任务？
2. 当前 `ai-collab-foundation` 应该怎么创建第一批任务？

## 1. 什么时候创建 Linear 任务

默认规则：

- 纯探索阶段可以先不建
- 一旦开始正式实现，就应该建

更具体地说，只要满足以下任一条件，就应创建 Linear 任务：

1. 开始写代码
2. 开始拆 milestone
3. 开始定义验收标准
4. 需要多人协作、回顾、排期或追踪
5. 需要把“聊天中的想法”变成“可验证的交付承诺”

## 2. 什么情况还可以不建

以下活动可以先不建 Linear：

1. 方向讨论
2. 产品定位探索
3. 竞品拆解
4. 低保真草图或参考图收集
5. PRD 和 roadmap 的首版起草

但一旦从“定义问题”切到“开始实施”，就不要继续停留在聊天驱动。

## 3. 当前项目是否属于空跑

不是。

原因是当前项目并非无依据推进，而是基于：

- PRD / roadmap
- 本地产品文档
- 已实现代码
- 已验证测试
- 已推送提交

但它也不是严格意义上的“从第一天起完全 Linear 驱动”。

更准确的描述是：

`文档驱动 + 代码驱动 + 本地 evidence 驱动`

现在已经到了必须切换成：

`Linear + 本地 evidence + 代码执行`

的阶段。

## 4. 当前应该怎么建第一批任务

建议按下面顺序：

1. 创建 milestone：`M1 · 本地配置生成器 MVP`
2. 录入 3 个已完成 issue：
   - `M1-01 AI 工作流诊断与首版主流程`
   - `M1-02 AI-OS 输出、预览与真实写出`
   - `M1-03 多客户端适配包与接入检查`
3. 为每个 issue 附上本地 `docs/linear/` 对应文件路径
4. 再创建下一批待开发 issue

## 5. 为什么要先录入已完成 issue

因为这些能力已经真实存在：

- 页面已实现
- 流程已打通
- 文件已可真实写出
- 多客户端适配包已可导出
- setup-check 已存在
- typecheck / build / e2e 已通过

如果不把这部分录入 Linear，后续 roadmap 会丢失历史上下文，团队也无法清楚知道第一阶段到底交付了什么。

4. `M1-07 权限边界与风险说明增强`
5. `M1-08 AI-OS 内容质量与可复用规则增强`
6. `M1-09 Setup Check 引导与最终落位优化`

## 7. 后续执行规则

从这一步开始，建议固定为：

1. 先创建或更新 Linear issue
2. 再在本地 `docs/linear/` 建立镜像说明
3. 然后开始实现
4. 完成测试、证据、人工验收记录后再标记 Done

不要再让功能只存在于聊天、commit 或临时记忆里。
